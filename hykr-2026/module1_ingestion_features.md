## Strongest features (7, ranked by engineering depth)

### 1. Dual-curve crypto transport for the AA fetch pipeline
**Brief-ready one-liner:** Each AA round-trip generates per-session X25519 (`crypto.generateKeyPairSync`) or, for mutual-fund FIPs that reject X25519, Curve25519 key material via an in-process BouncyCastle Java helper; the FIP's `KeyMaterial` is then combined with the saved private key to derive a session key + IV (`deriveKeyAndIV`) and decrypt the per-account `encryptedFI` blob with AES-GCM. Outbound requests carry a JWS detached signature (RFC 7797 `b64:false`); inbound responses are verified by resolving `kid` against a per-environment public-key cache.
**Evidence:** `services/ingestion/AA/aa-transport.js:211-286` (key material), `services/ingestion/AA/aa-transport.js:289-360` (RFC 7797 verifier), `services/ingestion/AA/fi-data-processor.js:223-278` (ECDH derive + AES-GCM decrypt + per-linkRef partial-failure persistence)
**Why this signals real engineering:** The code holds raw key material for a single fetch, isolates the BouncyCastle path behind a curve-by-consent-group switch, and persists `failedLinkRefs` separately from `decryptOk` so a reconciler can retry only the items that failed — not the whole session.

### 2. Request-scoped DI factory across multiple AAs (no mediator, no circular deps)
**Brief-ready one-liner:** `aa-factory.js` resolves an `aaId` from the request, constructs a fresh dependency bag — `AATransport(aaId)` → `ConsentService`, `SessionService`, `FIDataProcessor`, `FIFetcher`, `SessionStatusService`, `NotificationHandler` — and returns it; no service imports another service, and all cross-service coordination happens in the controller layer. The most recent decomposition (Feb 2026, commit `c279d44`) split the 2,063-line `StatusCheckManager` god-class into `fi-data-processor`, `fi-fetcher`, and `session-status.service`, which the factory now wires alongside the pre-existing transport/consent/session/notification services.
**Evidence:** `services/ingestion/AA/aa-factory.js:31-77`, `services/ingestion/AA/aa-transport.js:27-76` (provider-scoped axios client + shared `https.Agent({maxSockets:4, keepAlive:true})`), `services/ingestion/AA/index.js`
**Why this signals real engineering:** A factory call costs nothing because the dependency graph is shallow and acyclic; the `maxSockets:4` HTTPS agent is a deliberate hard cap on concurrent connections to AA providers shared across all per-request clients.

### 3. Webhook → fetch state machine with atomic queue acquisition
**Brief-ready one-liner:** Each FI status webhook is flattened into `accounts[]`, projected into a Map keyed on the safe-encoded `linkRefNumber`, and persisted with `$set` on `linkRefMappings.<safeKey>` while guarding against late `TIMEOUT`/`DENIED` notifications overwriting already-fetched data; the queue → fetching → idle transition uses `findOneAndUpdate({ fiFetchState: { $nin: ["QUEUED", "FETCHING"] }})` to ensure only one worker ever begins a fetch even under PM2 cluster mode.
**Evidence:** `services/ingestion/AA/notification-handler.js:75-106` (safeLinkKey + atomic $set with prior-fetched guard), `services/ingestion/AA/notification-handler.js:175-209` (late-terminal protection), `services/ingestion/AA/notification-handler.js:213-280` (atomic queue acquisition + recovery path to mark COMPLETED)
**Why this signals real engineering:** The handler reasons about *real* webhook pathology — Saafe/Finvu sending `EXPIRED`/`TIMEOUT` hours after data was already delivered — and refuses to downgrade a session that has fetched data, instead of trusting the latest notification.

### 4. Self-learning email extraction templates (`tpl:{domain}:{templateHash}`)
**Brief-ready one-liner:** Email bodies are normalised to a structural fingerprint by replacing amounts/dates/account/reference/time with placeholders, then SHA-256-hashed to a 16-char `templateHash`; on first LLM success, a per-field regex rule (amount, direction, merchant, accountLast4, reference) is reverse-engineered from the body and stored with `state: 'active'` in both Redis (`tpl:{domain}:{hash}`, no TTL) and MongoDB. Subsequent emails with the same fingerprint short-circuit the LLM. Templates degrade automatically: `state: 'degraded'` after >30% failure over ≥5 uses, `state: 'expired'` after >50% over ≥10. A burn-in sampling rate (50% until useCount<20, then 5%) drives shadow-LLM validation to catch silent drift.
**Evidence:** `services/email/extraction-template.service.js:45-67` (template hash), `services/email/extraction-template.service.js:148-358` (rule builders for amount/direction/merchant/accountLast4/reference), `services/email/extraction-template.service.js:614-650` (degrade/expire lifecycle), `services/email/extraction-template.service.js:661-699` (validation sampling)
**Why this signals real engineering:** The system gets cheaper per user over time without manual template authoring; the degrade/expire thresholds and burn-in sampling treat templates as live state, not config.

### 5. SIG_V1 cross-source transaction signatures (AA ↔ Email ↔ PDF dedup)
**Brief-ready one-liner:** Every transaction — regardless of source — is hashed into two SHA-256 signatures over a pipe-delimited canonical form: `localSignature` (includes `sourceNamespace`, for in-source idempotency) and `crossSignature` (omits source, for AA↔Email↔PDF dedup). Normalisation is deterministic: amount in minor units (paise) via `Math.round(Math.abs(amount)*100)`, UTC date via `moment().utc().format('YYYY-MM-DD')`, merchant lowercased with company suffixes (`pvt|ltd|inc|corp`) and payment terms (`payment|txn|transfer`) stripped, account key canonicalised to `CARD:1234:INR` from `****1234`/`XXXX1234`/`ending 1234` patterns. A V2 signature runs in shadow mode for observation under `featureFlags.SIGNATURE_V2_OBSERVE`.
**Evidence:** `utils/transaction-signatures.js:29-176`, `services/ingestion/AA/unified-transaction-ingester.js:106-162` (dual-signature observation), `services/email/email-ingestion.service.js:654-688` (email path uses same generator)
**Why this signals real engineering:** This is the load-bearing primitive that lets the same transaction reach the user once whether it arrived as an AA bank line, an HDFC SMS-via-email, or a credit-card PDF — and the V1/V2 shadow path proves the team rolls out signature changes safely.

### 6. Credit bureau pull: bulkhead → circuit breaker → cluster lock → idempotency
**Brief-ready one-liner:** A bureau pull is gated by a four-stage guard: (a) `Bulkhead(concurrency=2)` queue around the HTTPS call, (b) a process-wide `BureauCircuitBreaker` that only increments on infra codes (`HTTP_ERROR | NETWORK_ERROR | TIMEOUT | EMPTY_RESPONSE | IP_NOT_WHITELISTED`) and *not* per-user codes (`API_REJECTED | IDENTITY_NO_MATCH | PARSER_ERROR`), (c) a MongoDB-backed cluster lock at `bureau-pull:${userId}` with 90s TTL that fails fast (`BUREAU_LOCK_HELD`) rather than silently skipping, and (d) an idempotency key `${userId}:${consentVersion}:${YYYYMMDD-IST}` that catches both the pre-call check and the unique-index race (E11000). Transport retries use jittered exponential backoff; timeouts are *not* retried because the vendor still bills.
**Evidence:** `services/ingestion/credit-bureau/bureau-circuit-breaker.js:29-52`, `services/ingestion/credit-bureau/bureau-cluster-lock.js:33-68`, `services/ingestion/credit-bureau/credit-bureau.service.js:72-74` (IST-day idempotency key), `services/ingestion/credit-bureau/credit-bureau.service.js:198-269` (runPull lifecycle with idempotency-race fallback), `services/ingestion/credit-bureau/experian-idspay.client.js:25-105` (typed `BureauError`, bulkhead, jitter)
**Why this signals real engineering:** The breaker distinguishes "vendor is down" from "user has bad data" — a one-line policy that prevents one wrong PAN from tripping the breaker for everyone. The IST-day idempotency key is the right scope for a daily-quota API.

### 7. Encrypted PDF password resolver with per-bank candidate generators
**Brief-ready one-liner:** For HDFC and SBI credit-card statements, the resolver gathers identity (name parts via prefix-stripping of `MR.|MRS.|SHRI.`, DOB, PAN, cardLast4 — auto-discovered from prior email history if absent) and emits bank-specific candidate passwords (`HDFC: NAME4+LAST4 | NAME4+DDMM | NAME4+DDMMYYYY`; `SBI: DDMMYY+LAST4 | DDMMYYYY+LAST4`), tries each via `qpdf` under a 10s per-attempt and 30s total timeout, and on success encrypts and caches the winning password keyed on `{userId, bankCode, cardLast4}` so repeat statements decrypt instantly.
**Evidence:** `services/ingestion/pdf-password-resolver.js:41-94` (resolve flow + auto-discover cardLast4), `services/ingestion/pdf-password-resolver.js:341-427` (HDFC/SBI generators), `services/ingestion/pdf-password-resolver.js:96-138` (stored-password fast path), `models/pdf-password-config.model.js` (per-user encrypted credential)
**Why this signals real engineering:** The password space is encoded as bank-specific rules, not a brute-force loop; failed candidates are bounded by `MAX_RETRIES=3` and timeouts so a malformed PDF can't stall ingestion.

## Hidden differentiators — in code but NOT in current doc

Each of these is implementation work a reviewer would respect but the current §4 omits:

- **AAKeyStore with in-flight request de-duplication, bulk-fetch-by-notifier, positive (24h) / negative (10min) caching, and a 12h proactive refresh window** — concurrent requests for the same notifier wait on a single in-flight `Promise` (`services/ingestion/AA/aa-keystore.service.js:78-116`). - **LogGrouper batches noisy event streams into one summary log per 30s window** — `outgoing-api`, `fi-notification`, `webhook-incoming`, `aa-callback`, `credit-bureau-pulls`, `credit-bureau-warnings`, and 7 others, all flushed on graceful shutdown. - **Boot-phase log buffering** — `logger.info()` is temporarily overridden during startup so 20 scattered boot lines emit as one consolidated `Boot complete` line with a `steps` count.

- **`Bulkhead` and `CircuitBreaker` extracted as library primitives** under `lib/circuit-breaker.js` and `lib/bulkhead.js` — the bureau path is a thin policy subclass, and 138 bureau tests rely on this separation (`bureau-circuit-breaker.js:9-12` references the extraction PR).
