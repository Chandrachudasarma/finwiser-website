## Strongest features (7, ranked by engineering depth)

### 1. Per-scheme MF analytics engine — pure financial math on raw NAV time-series
**Brief-ready one-liner:** Computes the full risk/return surface for every tracked scheme directly from the daily NAV series — trailing returns (1M/3M/6M/1Y absolute and 2Y/3Y/5Y CAGR via binary search on a date-indexed array with a 30-day slippage tolerance), annualised volatility (√252 × σ of daily log returns), Beta and Jensen's Alpha against a per-scheme benchmark proxy on a 3-year window (`cov/var` on date-aligned series), Sharpe ratio computed from the geometric mean of all daily log returns *and* a secondary weekly-downsampled Sharpe to reduce microstructure noise (López de Prado), maximum drawdown over the full history, and an **Information Ratio with deflation `IR × (1 − 1/4N)`** on a fixed 3-year window so the IR is comparable across funds. Anomaly guards skip any single-day log return > log(1.30) (split detection) or weekly > log(1.50). Debt funds take an alternative path that skips the equity benchmark entirely and instead emits a SEBI-category-derived duration bucket plus a credit-risk flag (drawdown > 2× typical for that bucket).
**Evidence:** `services/ledger/market-data/analytics-compute.service.js` (945 lines, all formulas above), `test/ledger/market-data/analytics-compute.test.js` (16 `describe` blocks across all functions, incl. a `golden-file return accuracy` block: computed returns match Value Research / Morningstar within ±0.5%).
**Why this signals real engineering:** Industry-standard formulas are correctly implemented, plus subtle corrections most fintechs skip: deflation for IR estimation bias, weekly-downsampled Sharpe alongside daily, split-detection at the log-return level, and a separate code path for debt funds (where beta/alpha vs an equity index is mathematically meaningless).

### 2. Returns-based Style Analysis (Sharpe RBSA, 4-factor OLS)
**Brief-ready one-liner:** Decomposes each equity fund's behaviour into factor exposures (LargeCap / MidCap / SmallCap / Debt) via OLS regression of monthly returns against four index-fund proxies (Nifty 50, Nifty Midcap 150, Nifty Smallcap 250, HDFC Short Term Debt), solving `β = (XᵀX)⁻¹XᵀY` by Gaussian elimination on the normal equations, then floor-and-renormalising negative coefficients to produce non-negative weights summing to 1 (Morningstar Style Box approach). Uses a 60-month preferred / 36-month minimum window with an R² ≥ 0.80 *display gate* below which the model is considered too noisy to surface to users, and a 95% confidence band for style-drift detection rather than a fixed threshold.
**Evidence:** `services/ledger/market-data/style-analysis.service.js` (383 lines), `test/ledger/market-data/style-analysis.test.js`.
**Why this signals real engineering:** Detects style drift without needing holdings disclosure — a "Large Cap Fund" that regresses to 60/30/10 LC/MC/SC has drifted from its mandate. Hand-rolled OLS plus a principled non-negativity transform and a statistical gate on R², instead of a black-box library call.

### 3. Peer-ranking engine with category distribution stats
**Brief-ready one-liner:** Sunday-02:00-IST cron iterates every SEBI category × planType (Direct ranked separately from Regular, because the 0.5–1.5% structural expense-ratio gap makes cross-plan ranks unfair), computes 1Y/3Y/5Y trailing returns from `nav_history`, sorts descending, assigns `rank` and `percentile = round((n−1−i)/(n−1) × 100)`, and upserts both per-scheme percentile fields *and* a per-category distribution document (median, mean, p25, p75, min, max via linear-interpolation percentile matching Excel `PERCENTILE.INC`). A `MIN_SCHEMES_FOR_RANKING = 10` constant gates the UI "reliable ranking" caveat (not enforced inside the ranking computation itself); NFOs with <1Y history get an explicit `null` 1Y rank (not silently excluded); the file's design header treats 3Y as the primary display metric for ELSS (3-year lock-in) — applied UI-side, since the backend computes 1Y/3Y/5Y uniformly; a bounded in-memory benchmark-NAV cache prevents re-fetching the ~30 unique benchmarks while computing rolling beat rates for top-10 schemes per category.
**Evidence:** `services/ledger/market-data/peer-ranking.service.js` (587 lines), `jobs/weekly-peer-ranking.job.js` (cron `30 20 * * 6`), `models/ledger/category-stats.model.js`, `test/ledger/market-data/peer-ranking.test.js` (27 tests).
**Why this signals real engineering:** Most peer ranks in the market mix Direct + Regular and silently drop NFOs. This pipeline gets the cohort definition right, persists the full distribution (not just the rank), and bounds memory during the cross-category batch.

### 4. Dirty-marking analytics cache with in-flight Promise dedup
**Brief-ready one-liner:** Daily NAV append marks every affected `analytics_cache` document `isDirty: true`; the next `getAnalytics(amfiCode)` call sees the flag, triggers a single `computeAndCache()`, and stores the result with `isDirty: false`. A second-level `Map<amfiCode, Promise>` coalesces concurrent callers — page-load endpoints hitting `getAnalyticsBatch()` for the same code share one in-flight Promise instead of stampeding the compute path. No TTL on the in-flight map (the dirty flag is the staleness manager); a lazy sweep enforces `MAX_INFLIGHT_ENTRIES = 500` to bound memory.
**Evidence:** `services/ledger/market-data/analytics-compute.service.js` (`_analyticsInflight`, `getAnalytics`, `computeAndCache`), `models/ledger/analytics-cache.model.js` (`markDirty`, `initDirty`, `getDirty` statics + indexed `isDirty` flag), `test/ledger/market-data/inflight-dedup.test.js`.
**Why this signals real engineering:** Two orthogonal stampede-prevention layers (dirty-flag + Promise coalescing) used together, with each layer's failure mode separately bounded. Survives multi-process PM2 fork mode because the dirty flag lives in Mongo, not memory.

### 5. XIRR with transaction-coverage tiering and synthetic-remainder cash flows
**Brief-ready one-liner:** Newton-Raphson solver for the NPV root with a bisection fallback that's mathematically guaranteed to converge whenever one negative and one positive cash flow exist. When AA-supplied transaction history doesn't cover the user's full cost basis (common for older folios where the FIP only returns the last 6–12 months), the engine fills the gap with a *synthetic* lump-sum cash flow dated to the earliest available transaction and returns an accuracy tier — `full` (coverage ≥ 95%), `partial` (10–95%), `synthetic` (<10%) — so the UI can disclose what it's looking at. A portfolio-level 80% gate suppresses XIRR + benchmark comparison entirely when synthetic dates dominate, instead of silently presenting a misleading IRR. Cash-flow date uses `transactionDate` (allotment date), matching CAMS / KFintech CAS statements for tax-side reconciliation.
**Evidence:** `services/ledger/portfolio/xirr.service.js` (644 lines), `test/ledger/portfolio/xirr.test.js` (53 tests), design doc `docs/decisions/2026-03-25_xirr-portfolio-returns-design.md`.
**Why this signals real engineering:** Both numerical-method correctness (NR + bisection fallback, divergence guards `rate ∈ [−0.99, 10]`) *and* the much-harder product-side honesty about data quality. The 80% gate is the engineering choice most competitors get wrong.

### 6. Per-lot capital-gains tax engine — two-tier model with FY 2025-26 three-regime classifier
**Brief-ready one-liner:** Tier 1 reconstructs lots per-transaction for the last 12 months (or 36 for ELSS lock-in detection); Tier 2 collapses everything older into a single aggregate lot derived from the AA-supplied `costValue` / `closingUnits` — because anything older than 12 months on an equity fund is guaranteed LTCG and per-lot precision adds zero accuracy beyond aggregate cost basis. Lots are classified against the FY 2025-26 three-regime model: **EOF** (Equity-Oriented Fund — 20% STCG s.111A, 12.5% LTCG s.112A, ₹1.25 L annual exemption), **SMF** (Specified MF s.50AA — slab rate always, no LTCG concept), **OMF** (Other MF — slab STCG, 12.5% LTCG s.112). Pre-2018 grandfathering rule applied. Reconciliation deltas between AA-reported units and reconstructed lots are tracked (`<0.01` unit = reconciled, `<1.0` = reconciled with warning).
**Evidence:** `services/ledger/portfolio/lot-tax-computation.service.js` (893 lines), `config/ledger/tax-regime-classifier.js`, `test/ledger/portfolio/lot-tax-computation.test.js` (79 tests), design docs `2026-03-23_per-lot-tax-computation-design.md` and `2026-04-03_tax-rule-fy2025-26-update-design.md`.
**Why this signals real engineering:** The "only the last 12 months need per-lot precision" insight is the kind of observation that comes from actually doing the math. The three-regime classifier matches the post-Finance Act 2024 statute rather than the legacy debt-MF rules most apps still ship.

### 7. AA × Credit-Bureau dedup with PAN-aware join key
**Brief-ready one-liner:** Bureau credit-card rows are joined to AA credit-card holdings on `(normalised lender, last-4 digits)` with three categories of carve-out: (a) when the bureau row's `holderPan ≠ consentPan`, the row is a joint-account or supplementary card for someone else and is **not** deduped (kept counted); (b) insurance-subsidiary lenders that share a brand prefix (HDFC ERGO, ICICI Lombard) are excluded from the lender-canonicalisation map and never match a bank CC; (c) closed-account status codes (82/83/84/85) are dropped entirely. Matched bureau rows are tagged `dedupedBy: 'AA'` and excluded from the recomputed total but retained in the per-account list so the UI can show "Already linked via AA." Pure read operation — no paid third-party write side-effects on read paths (CQRS boundary).
**Evidence:** `services/ledger/unified-portfolio.service.js` (`dedupBureauCreditCardsAgainstAA`, lines 300–398), `services/ingestion/credit-bureau/experian-account-types.js` (`normalizeLender`, `isCreditCardType`), design `docs/.../unified-portfolio §10` (12 edge cases enumerated).
**Why this signals real engineering:** Real-world data has edge cases that naïve `(lender, last4)` joins miss — the PAN-mismatch carve-out alone catches a class of false positives that would otherwise undercount a family's combined credit exposure. The 12-edge-case enumeration in the design doc shows this was driven by data, not pattern-matching.

## Hidden differentiators — in code but NOT in current doc

- **Implicit Expense Ratio** — `peer-ranking.service.js:471` (`computeImplicitExpenseRatios`) pairs every Regular-plan scheme with its Direct counterpart via a normalised `(amc, schemeName)` key (strips "Direct Plan" / "Regular Plan" tokens) and stores the **actual** Direct−Regular 1Y-return delta as `implicitExpenseRatio` on the Regular scheme's `analytics_cache` document, plus the Direct AMFI code for a one-tap switching nudge. Captures hidden distributor charges that nominal TER does not. Currently invisible in the brief.
- **Information Ratio (raw + deflated)** — `computeInformationRatio` returns both the raw IR and `rawIR × (1 − 1/4N)` deflated for small-sample estimation bias, on a fixed 3-year window for cross-fund comparability. Only computed for EXACT or PROXY benchmarks (skipped for FALLBACK / sectoral funds where IR would be misleading). The deflation is the kind of detail a finance-savvy reviewer will spot.
- **Rolling 12-month beat rate, quarterly-stepped** — `computeRollingReturns` deliberately uses a 63-trading-day step (quarterly) rather than daily-stepped windows, citing autocorrelation explicitly: "5 years of daily-stepped rolling returns gives ~1,000 points sharing 11 months of data — that's illusory sample size. Quarterly stepping gives ~16 truly independent observations." This is the López de Prado fix.
- **5-pillar Portfolio Score with floor rule** — `services/ledger/portfolio-score.service.js` (1,507 lines): DebtHealth 10% + Performance 25% + WealthMix 25% + Suitability 20% + Protection 20%, with a **hard cap of 45 if any pillar < 20** (prevents a "weighted average looks fine" pathology when one dimension is broken). 30-second TTL in-flight cache; cross-module data only via `interfaces/ledger.interface.js` / `analysis.interface.js`. The interface boundary is significant — this is how M2 stays a "facts" module without leaking into M4 judgments.
- **HHI concentration with effective-positions inverse** — `getConcentration` in `portfolio-intelligence.service.js:357` computes `HHI = Σwᵢ²` over MF weights, `effectivePositions = 1/HHI`, plus separate AMC-level concentration map and a top-2-weight scalar. Currently appears as "(HHI)" in parentheses with no math.
- **Anomaly guards on log returns** — `MAX_DAILY_LOG_RETURN = log(1.30)` and `MAX_WEEKLY_LOG_RETURN = log(1.50)` skip suspected splits/consolidations / data errors. Counter is persisted as `skippedNavDays`; `navDataSuspect` flag set when > 5 anomalies, which excludes the fund from peer ranking and IR.
- **Weekly-downsampled Sharpe** alongside daily Sharpe — explicitly to reduce bid-ask-bounce / stale-NAV-on-holiday noise. Stored as `weeklySharpe` on every analytics doc.
- **NAV pipeline architecture** — 23:30 IST cron parses AMFI `NAVAll.txt` → upserts `instrument_master` (~14K MF schemes) → `appendDailyNavs` bulk-inserts today's NAV via `insertMany({ ordered: false })` (handles dup-keys natively) → marks affected `analytics_cache` dirty. `mfapi.in` history backfill uses 300 ms inter-request politeness delay and an `AbortController` hard timeout at 20 s.
- **Benchmark resolution tiering** — `benchmark-proxy.service.js` resolves `EXACT` (proxy tracks index <0.5% TE) / `PROXY` (correlated substitute) / `FALLBACK` (no scheme data). IR is only shown for EXACT + PROXY; rolling beat rate is only computed for EXACT + PROXY; FALLBACK gets a UI caveat instead of a misleading number.
- **Insurance protection score** — `protection-assessment.service.js:36-42`: weighted Life 40% / Health 30% / Property 20% / Vehicle 10%, with **three explicit data states** (`not_checked` → `null`, `verified_none` → 0, `verified_partial/full` → computed). Coverage expressed as months-of-expenses for user-comprehensibility. Need formulas: Life = 10× annual income; Health = max(₹10L, 20% annual income).
- **Manual assets layer** — `models/ledger/manual-asset.model.js` lets users add real estate, gold, private equity, etc. that AA can't see, and these flow into `getPortfolioIntelligence()` and the WealthMix pillar. Currently invisible in the brief.
